1. 캠퍼스 온도 집계 — AI 감정 분석 + 양방향 온도 보정

[양방향 온도 보정 로직]
Java
private double calculateBidirectionalAdjustment(
    double currentTemp, int postCount, int hour) {

  PostActivityLevel activityLevel =
      postActivityAnalyzer.analyzeCurrentActivity(postCount);
  double baseRate = activityLevel.getAdjustmentRate();
  double protection = getTemperatureProtectionFactor(
      currentTemp, activityLevel.isIncreasing());

  TemperatureGuideline guideline =
      guidelineConfig.getGuideline(hour);
  double timeMult = activityLevel.isIncreasing()
      ? guideline.getIncreaseMultiplier()
      : guideline.getDecreaseMultiplier();

  return currentTemp * (1 + baseRate * protection * timeMult);
}
[설계 포인트]
  • 활동량 분석 → 상승/하강 판단
  • 보호 계수로 극단값(0°/100°) 접근 시 변화폭 자동 감소
  • 시간대별 가이드라인으로 낮/밤 보정 배율 분리
  • 게시글 활동을 하나의 서비스 지표(캠퍼스 온도)로 시각화
  • 시간대별 보정으로 극단적 온도 변동 방지

public String uploadFile(MultipartFile file) {
  String contentType = file.getContentType();
  String folder = determineFolder(contentType);
  String key = folder + "/contents/"
    + LocalDate.now().format(
        DateTimeFormatter.ofPattern("yyyy/MM/dd"))
    + "/" + generateFileName(
        file.getOriginalFilename());

  PutObjectRequest req = PutObjectRequest.builder()
      .bucket(bucketName).key(key)
      .contentType(contentType).build();
  s3Client.putObject(req,
    RequestBody.fromInputStream(
      file.getInputStream(), file.getSize()));
  return baseUrl + key;
}
[저장 구조 분리 결과]
  • 영속 데이터 → MariaDB
  • 캐시 / 실시간 조회 → Redis
  • 이미지·녹음 파일 → S3 (Content-Type별 폴더)

1. 계좌 데이터 정합성 — 외부 API 동기화 (Reconciliation)
로컬 DB의 계좌 정보와 외부 금융 API 데이터 사이에 불일치가 발생하면
분석, 예산, 목표 화면 전체의 일관성이 깨지는 문제가 있었습니다.
외부 API 기준으로 비교·동기화하고, 삭제된 계좌는 비활성 처리하는 reconcile 구조를 구현했습니다.
[계좌 Reconciliation 로직]
Java
@Transactional
public List<Account> execute(Integer userId) {
  List<Account> linked =
    accountRepo.findAllLinkedByUserId(userId);
  User user = findUserUseCase.execute(userId);

  // 외부 API에서 전체 계좌 조회
  Map<String, AccountItem> external =
    apiClient.inquireAccounts(user.getUserKey())
      .accounts().stream()
      .collect(Collectors.toMap(
        AccountItem::accountNo, identity()));

  return linked.stream()
    .filter(acc -> reconcile(acc, external))
    .toList();
}

private boolean reconcile(Account acc,
    Map<String, AccountItem> external) {
  AccountItem ext = external.get(acc.getAccountNo());
  if (ext == null) {
    acc.markInactive(); return false;
  }
  acc.sync(ext); acc.markActive();
  return true;
}
[Reconciliation 포인트]
  • 외부 API 기준 Map 구성 → O(1) 비교
  • 존재하지 않는 계좌 → markInactive (삭제 대신 비활성)
  • 메타데이터 변경 → sync로 일괄 갱신
  • @Transactional로 부분 실패 방지
  • 홈, 분석, 목표 화면이 동일한 데이터 기준을 바라보도록 정합성 확보
  • 외부 API 장애 시에도 기존 데이터 보존 (비활성 처리)
2. 예산 배분 알고리즘 — 월간 수입 → 주간 배분 + 부족분 조정
단순 지출 분석만으로는 사용자가 행동으로 옮기기 어려웠습니다.
월간 수입을 주 단위로 분배하면서 고정비, 목표 저축, 여유 자금을 모두 반영하는 다단계 배분 로직을 구현했습니다.
[예산 배분 핵심 흐름]
Java
public AllocationResult allocateBudget(
    User user, LocalDate start,
    Long monthlyIncome,
    List<EssentialExpenseMonth> essentials,
    List<Plan> activePlans) {

  // 1. 정규화
  long income = normalizeAmount(monthlyIncome);
  // 2. 주 프레임 생성
  List<WeekFrame> weeks =
    buildWeekFrames(start, allocationEnd);
  // 3. 월 예산 → 주 분배
  Map<LocalDate,Long> incomeByWeek =
    allocateMonthlyIncome(start, end,
      income, weeks);
  // 4. 고정비 할당
  List<EssentialAlloc> essentialAllocs =
    buildEssentialAllocations(...);
  // 5. 부족분 조정
  applyShortageAdjustments(weeks,
    budgetWeeks, incomeByWeek,
    essentialByWeek, planAllocs);

  return new AllocationResult(...);
}
[배분 로직 구조]
  • 월간 수입 → 주 단위 균등 분배
  • 고정비(필수 지출) 우선 차감
  • 목표 저축 계획 반영
  • 부족분 발생 시 자동 조정(Shortage Adjustment)

[핵심 판단]
  • 분석 결과 → 계획 수립 → 실행으로 이어지는 흐름 설계
  • 조회 중심이 아닌 실행 중심 서비스로 전환
  • 분석 결과가 주간 예산과 목표 계획으로 직접 연결
  • 서비스 흐름을 조회 → 실행 중심으로 확장
