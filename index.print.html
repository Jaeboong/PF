1. 캠퍼스 온도 집계 설계
게시글 목록만으로는 분위기를 직관적으로 전달하기 어려웠기 때문에, 활동량과 감정 분석 결과를 연결한 온도 지표를 설계했다.
핵심 로직
private double calculateBidirectionalAdjustment( double currentTemp, int postCount, int hour) { PostActivityLevel activityLevel = postActivityAnalyzer.analyzeCurrentActivity(postCount); double baseRate = activityLevel.getAdjustmentRate(); double protection = getTemperatureProtectionFactor( currentTemp, activityLevel.isIncreasing()); TemperatureGuideline guideline = guidelineConfig.getGuideline(hour); double timeMult = activityLevel.isIncreasing() ? guideline.getIncreaseMultiplier() : guideline.getDecreaseMultiplier(); return currentTemp * (1 + baseRate * protection * timeMult); }
설계 포인트
  • 활동량 분석으로 상승/하강 방향 판단
  • 0도, 100도에 가까울수록 보호 계수로 변화폭 축소
  • 시간대별 가이드라인으로 낮, 밤 보정 배율 분리
  • 지표가 급변하지 않도록 사용성 중심으로 완충
Result
  • 홈 화면에서 캠퍼스 분위기를 즉시 읽을 수 있는 서비스 지표를 확보
  • 게시글이 적은 시간대에도 지표가 흔들리지 않도록 안정성 확보
2. S3 업로드 경로 자동 분류
이미지와 녹음 파일을 한 경로에 두면 운영과 검색이 복잡해지기 때문에, Content-Type 기반 폴더 분리 전략을 적용했다.
핵심 로직
public String uploadFile(MultipartFile file) { String contentType = file.getContentType(); String folder = determineFolder(contentType); String key = folder + "/contents/" + LocalDate.now().format(DateTimeFormatter.ofPattern("yyyy/MM/dd")) + "/" + generateFileName(file.getOriginalFilename()); PutObjectRequest req = PutObjectRequest.builder() .bucket(bucketName).key(key).contentType(contentType).build(); s3Client.putObject(req, RequestBody.fromInputStream(file.getInputStream(), file.getSize())); return baseUrl + key; }

Project 02 / Service Flow
모아 Service Structure
분석 결과를 실행 흐름으로 연결한 금융 서비스 구조
Outcome
  • 분석 결과 화면에서 끝나지 않고 목표 계획 수립, 주간 예산 설정, 홈 자유소비 관리까지 이어지는 실행 흐름을 구성했다.
  • 계좌 동기화 이후 분석, 예산, 목표 화면이 동일한 계좌 기준을 보도록 데이터 기준점을 통일했다.
  • 목표 충돌 시 부족분을 우선순위가 낮은 목표부터 조정해 금액 변화 이유를 설명 가능한 구조로 설계했다.
Why This Structure
  • Spring Boot를 메인 API 서버로 두어 계좌, 거래, 목표, 예산처럼 트랜잭션이 중요한 금융 도메인을 일관된 계층으로 관리했다.
  • FastAPI는 분석 응답처럼 계산 로직 변경이 잦은 영역을 분리할 때 유리해 금융 도메인 API와 분석 파이프라인의 책임을 나누는 구조에 적합했다.
  • PostgreSQL과 Docker를 활용해 거래, 목표, 예산 관계와 팀 실행 환경을 안정적으로 맞췄다.
Execution Flow
1. Sync 계좌 연동 후 외부 금융 API 기준으로 계좌와 거래 데이터를 동기화한다.
2. Analyze 최근 거래 흐름을 집계하고 소비 패턴을 해석해 사용자 상태를 보여준다.
3. Plan 목표 설정과 우선순위 입력을 받아 예산과 저축 계획을 계산한다.
4. Act 주간 예산과 자유소비 금액을 홈 화면까지 연결해 사용자가 실제 행동으로 옮길 수 있게 한다.
What I Learned
금융 서비스는 계산식 자체보다도, 어떤 데이터를 기준으로 같은 결과를 반복해서 보여줄 수 있는지가 더 중요했다. 모아에서는 그 기준점을 계좌 동기화와 예산 배분 로직으로 고정했다.

모아 Account Reconciliation
외부 API 동기화와 데이터 기준점 통일
1. 계좌 데이터 정합성 문제
로컬 DB의 계좌 정보와 외부 금융 API 데이터가 어긋나면, 분석과 예산, 목표 화면 전체가 서로 다른 기준을 보게 된다.
Reconciliation Logic
@Transactional public List execute(Integer userId) { List linked = accountRepo.findAllLinkedByUserId(userId); User user = findUserUseCase.execute(userId); Map external = apiClient.inquireAccounts(user.getUserKey()).accounts().stream() .collect(Collectors.toMap(AccountItem::accountNo, identity())); return linked.stream() .filter(acc -> reconcile(acc, external)) .toList(); } private boolean reconcile(Account acc, Map external) { AccountItem ext = external.get(acc.getAccountNo()); if (ext == null) { acc.markInactive(); return false; } acc.sync(ext); acc.markActive(); return true; }
Design Points
  • 외부 API 기준으로 Map을 구성해 계좌 비교를 O(1)로 처리
  • 삭제 대신 비활성 처리로 일시 오류에도 계좌 이력을 보존
  • @Transactional로 부분 실패와 불완전 동기화를 방지
  • 분석, 예산, 목표가 모두 같은 계좌 기준점을 보도록 보장
Why It Matters
  • 금융 서비스는 계산 로직보다 먼저 어떤 데이터를 기준으로 읽고 쓰는지가 중요하다.
  • 동기화가 어긋나면 화면별 수치가 다르게 보이기 때문에 사용자 신뢰가 깨진다.
Result
  • 동기화 이후 홈, 분석, 목표가 동일한 계좌 목록을 기준으로 계산되도록 구조를 통일
  • 삭제된 계좌를 즉시 제거하지 않아 외부 API 오류 상황에서도 안정적으로 복구 가능
Service Impact
계좌 정합성은 모아의 모든 추천과 예산 배분 로직의 출발점이다. 이 기준점을 먼저 고정했기 때문에 이후 분석 결과와 계획 수립 흐름을 일관된 사용자 경험으로 연결할 수 있었다.
